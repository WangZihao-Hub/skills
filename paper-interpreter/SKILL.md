# Paper Interpreter — 学术论文深度结构化解读

## 概述

本技能用于对任意学科的学术论文（包括但不限于：计算机科学、人工智能、自然语言处理、计算机视觉、系统与网络、理论计算机、生物信息学、认知科学等）进行结构化深度解读。输出为一个独立的、自包含的HTML文件，读者无需任何外部依赖即可在浏览器中打开阅读。

## 核心设计理念

论文解读不同于论文翻译或简单摘要。一次高质量的解读应做到：

1. **忠于原文** — 准确传达论文的核心信息，不曲解、不夸大
2. **超越原文** — 站在更广阔的领域视角审视论文，点出作者没说的、没说透的
3. **结构化呈现** — 读者可以按需跳转到感兴趣的维度，不必通读全文
4. **可操作** — 读者读完后知道这篇论文值不值得精读、怎么复现、能用在哪儿

## 第一阶段：准备工作

### 1.1 确认论文内容来源

根据用户提供内容的方式，采取不同的策略：

- **用户直接粘贴全文**（最常见）→ 直接进入解读流程
- **用户提供PDF文件** → 优先使用PDF skill提取文本，如果PDF skill不可用或提取失败，要求用户粘贴文本
- **用户只给了标题/DOI/链接** → 在网上搜索该论文的摘要和公开信息，并告知用户"基于公开摘要解读，深度有限，建议提供全文以获得更全面的分析"
- **用户给了不完整的片段** → 明确告知缺失了哪些关键部分（方法？实验？），要求补充，同时基于现有内容给出初步分析

### 1.2 收集环境资源（重要：寻找相关Skill）

**在开始解读之前，必须先扫描当前环境中可用的所有 Skill**，判断哪些可能对论文解读或HTML生成有帮助：

```
# 在开始解读前，检查 available_skills 列表，判断哪些 Skill 可以增强本次输出：

1. 如果有 PDF skill → 论文是PDF时优先使用它提取文本
2. 如果有数据可视化相关 skill → 借鉴其图表设计风格用于维度10的可视化
3. 如果有 PPT/演示相关 skill → 借鉴其排版和配色方案增强HTML视觉效果
4. 如果有代码/公式渲染相关 skill → 借鉴其代码块和公式的呈现方式
5. 如果有领域专项 skill（如生物信息、金融等）→ 如果论文领域匹配，参考其专业术语和分析框架
6. 如果有 deep-research skill → 如果论文信息不足，可以使用它搜索补充背景资料
```

**如何使用找到的Skill：**
- 不要直接调用无关的skill，而是**阅读其 SKILL.md 或 assets 文件**，提取其中可用于增强HTML输出质量的CSS样式、布局模式、图表绘制方法
- 如果某个skill的模板/样式与本报告的某个部分高度相关，直接借鉴其最佳实践
- 例如：如果 `guizang-ppt-skill` 可用，阅读它的 assets，借鉴其大字报数据展示和网格布局的设计思路，用于维度5（实验结果）的数据呈现
- 例如：如果有任何包含 chart/diagram 样式的skill，借鉴其纯CSS图表的实现方式

**将找到的资源记录下来，在第三阶段生成HTML时主动使用。**

### 1.3 判断论文类型与调整解读侧重

不同领域的论文，解读时应有不同的侧重。在开始12维度分析前，先判断论文类型：

| 论文类型 | 典型特征 | 解读侧重 |
|---------|---------|---------|
| **方法型**（ML/CV/NLP主流） | 提出新模型/算法，在benchmark上对比SOTA | 方法论（维度4）和实验（维度5）是重点，批判性（维度9）应关注是否只在特定数据集上好使 |
| **系统型**（Systems/DB/OS） | 设计新系统/架构，强调性能/可扩展性 | 技术路线（维度4）中架构设计是重点，应画出架构图（维度10），关注工程trade-off |
| **理论型**（ML Theory/Math） | 定理证明、收敛性分析、泛化界 | 公式推导逻辑是重点，应拆解关键假设和证明思路（维度4），关注假设的合理性（维度9） |
| **经验/调查型**（Survey/Benchmark） | 大规模实验对比、领域综述 | 实验设计公平性（维度5）和覆盖面（维度9），贡献是否有insight而不只是跑实验 |
| **应用型**（Bio/Med/Geo等+AI） | 将AI方法应用于具体领域问题 | 领域问题（维度2+3）要讲清楚，方法创新可能不大但领域价值大（维度6+9关注跨学科意义） |

### 1.4 设定解读深度等级

根据用户意图（显式或隐式），判断期望的解读深度：

- **快速概览**：用户说"大概看看"/"扫一眼" → 每个维度2-4句话，HTML简洁
- **标准解读**（默认）：用户说"解读"/"分析" → 每个维度2-4段，HTML完整
- **深度精读**：用户说"仔细分析"/"精读"/"组会汇报" → 每个维度详细展开，HTML更丰富

## 第二阶段：按12个维度逐一深度解读

对每个维度的要求不只是"列出"，而是"理解、提炼、重组、批判"。以下是每个维度的详细操作指南：

---

### 维度1：元信息

**目标**：让读者一眼知道"这是谁、何时、在哪发表的、什么级别"。

**必须提取的信息**：
- 完整标题（含副标题）
- 所有作者的全名及其所属机构（至少写出第一作者和通讯作者的机构）
- 发表年份
- 发表的会议或期刊全称（如 CVPR 2024 → "IEEE/CVF Conference on Computer Vision and Pattern Recognition (CVPR) 2024"）
- CCF推荐级别 / SCI分区 / CORE排名（如果是计算机领域，推断或搜索其级别）
- DOI链接（如有）、arXiv ID（如有）
- 是否获奖（Best Paper / Oral / Spotlight 等荣誉）
- 引用量（如果已知）

**HTML呈现方式**：hero区域集中展示 + 可点击的链接

---

### 维度2：研究背景与动机

**目标**：回答"这个领域之前是什么状态？为什么需要这篇论文？"

**必须覆盖的要点**：
- 该研究所属的具体子领域（不要只说"AI领域"，要精确到"基于扩散模型的可控图像生成"这种粒度）
- 该子领域的简要演进脉络（里程碑工作2-4个，点出每个的贡献和不足）
- 现存的2-3个核心痛点（技术瓶颈、成本问题、性能上限等），越具体越好
- 本文针对的是其中哪个痛点？为什么之前的方法解决不了？
- 如果论文提出了一个新兴问题（之前没人做过），说明这个问题的来源和重要性

**写作要求**：
- 用"总→分→总"结构：先一句话概括背景，再展开细节，最后一句话点出本文的切入点
- 避免泛泛而谈（"近年来AI发展迅速" 这种话废掉，直接说具体是什么发展了）

**HTML呈现方式**：段落文字 + 可选的时间线或痛点列表

---

### 维度3：核心问题/目标

**目标**：精确表述"论文到底要解决什么问题"。

**必须覆盖的要点**：
- 用一句话精确陈述核心研究问题（Research Question）
- 如果有多个子问题，逐一列出
- 研究假设（Hypothesis）——论文隐含或明示的前提假设
- 预期目标——论文声称要达成的效果（定性和定量）
- 问题的输入-输出形式化定义（如果适用）：给定什么输入，期望得到什么输出

**注意区分**：
- "研究问题" ≠ "方法" — 问题是"要解决什么"，方法是"怎么解决"
- 例如：问题= "如何在低资源场景下进行高质量机器翻译"，方法= "提出了一种基于对比学习的跨语言预训练策略"
- 如果论文混淆了问题和方法的界限，在批判性思考（维度9）中指出

**HTML呈现方式**：醒目的问题陈述块 + 子问题编号列表

---

### 维度4：方法论/技术路线

**目标**：让读者在不读原论文的情况下，理解"这个方法是怎样工作的"。

**这是全篇解读中最长的维度，必须详细展开。**

**必须覆盖的要点**：

**(A) 总体架构概述**
- 方法的整体框架（用文字描述，HTML中配合CSS流程图）
- 输入→各组件→输出的数据流
- 训练阶段 vs 推理阶段的区别（如果有）

**(B) 核心组件逐一拆解**
- 每个关键模块的作用、输入、输出、设计动机
- 模块之间的依赖关系
- 为什么这样设计而不是那样设计（设计决策的推理）

**(C) 关键公式**
- 核心公式（不超过5个），每个公式解释其含义和每个符号的含义
- 公式之间的逻辑关系
- 损失函数的设计动机（每一项惩罚什么、鼓励什么）

**(D) 训练策略**
- 优化器、学习率调度、batch size、epoch数
- 训练技巧：数据增强策略、正则化方式、预训练权重来源
- 是否有两阶段/多阶段训练

**(E) 创新点提炼（对比现有方法）**
- 用表格列出本文方法 vs 最相关的2-3个现有方法的差异
- 明确指出：哪些是全新的设计，哪些是借鉴/改进，哪些是完全复用的

**(F) 算法伪代码（如果论文提供了或可以合理推断）**
- 在HTML中使用 `<pre><code>` 块呈现
- 伪代码应是可读的中文注释 + 关键步骤

**HTML呈现方式**：CSS绘制的架构流程图 + 组件分块说明 + 公式代码块 + 对比表格

---

### 维度5：实验设计与结果

**目标**：回答"这个方法到底好不好？好在哪？不好在哪？"

**必须覆盖的要点**：

**(A) 实验设置**
- 使用的所有数据集（名称、规模、领域、划分方式）
- 评估指标（为什么选这些指标？各指标的侧重）
- 所有对比的基线模型（分类：经典方法/最近SOTA/本文变体）
- 硬件环境（GPU型号、数量、显存）
- 实现框架（PyTorch / TensorFlow / JAX 等）

**(B) 主要结果**
- 用表格呈现核心对比实验的完整结果
- 标注最优值和次优值（加粗/颜色）
- 分析：在什么任务/指标上提升最显著？在什么上提升不大甚至变差？
- 提升幅度是否具有实际意义（不仅仅是统计显著）？

**(C) 消融实验**
- 每个消融实验的设置、目的、结论
- 哪个组件对性能影响最大？哪个几乎没用？
- 如果消融实验不完备，在维度9中指出

**(D) 参数敏感性**
- 关键超参数的变化对性能的影响
- 方法对哪些参数敏感、对哪些不敏感（敏感的→调参重要，不敏感的→方法鲁棒）

**(E) 可视化分析**
- 如果有特征可视化/注意力图/case study等定性分析，描述其发现
- 定性分析的结论是否和定量结果一致？

**HTML呈现方式**：多个对比表格（斑马纹、hover高亮）+ 柱状图/条形图（CSS绘制）+ 关键数字大字报

---

### 维度6：主要贡献

**目标**：简洁有力地总结"这篇论文凭什么被接收"。

**写作要求**：
- 以3-5条要点的形式列出
- 每条贡献用一句话标题 + 两句话展开
- 区分"核心贡献"和"次要贡献"
- 区分"理论贡献"和"实践贡献"
- 如果某项贡献存疑（声称很大但实际一般），在维度9中批判

**贡献的常见类型**：
- 新方法/新架构 → 方法贡献
- 新理论/新定理 → 理论贡献
- 新数据集/新benchmark → 资源贡献
- 新发现/新insight → 知识贡献
- 新SOTA → 实践贡献

**HTML呈现方式**：卡片网格（contrib-grid），每条贡献一张卡片

---

### 维度7：讨论与局限性

**目标**：忠实转述作者自己的反思和保留意见。

**必须覆盖的要点**：
- 作者明确承认的所有局限性（逐条列出）
- 每条局限性的严重程度评估
- 方法的适用范围边界（什么场景下这个方法不应该被使用？）
- 作者是否坦诚？有没有"避重就轻"？（如有，在维度9中展开）

**HTML呈现方式**：段落 + 关键局限用warning样式的alert框突出

---

### 维度8：未来工作建议

**目标**：给读者提供后续研究方向的参考。

**必须覆盖**：
- 论文中作者明确提出的后续方向
- 基于论文局限性的合理延伸（作者没提但自然能想到的）
- 基于领域发展趋势的推测（这个方向还能走多远？）
- 如果读者想在这个方向做后续工作，最值得做的1-2件事是什么？

**HTML呈现方式**：编号列表 + 每条方向的简要理由

---

### 维度9：批判性思考

**目标**：这是区分"会读论文的人"和"只会翻译论文的人"的关键维度。

**批判的11个角度（逐一检查）：**

1. **动机是否成立？** — 论文声称要解决的问题是否真实存在？是否值得解决？有没有可能是"为了发论文而造出来的问题"？
2. **假设是否合理？** — 方法依赖的前提假设在实际中是否成立？如果假设被违反会怎样？
3. **方法是否过拟合特定设置？** — 这个方法的成功是否依赖于特定的数据分布、特定的评估协议？换一个数据集/指标还会好吗？
4. **对比是否公平？** — 基线模型是否调参充分？是否用了相同的预训练数据？作者有没有在对比中"放水"？
5. **提升是否显著？** — 1个点的提升到底是实质突破还是随机波动？有没有做显著性检验？
6. **消融是否充分？** — 有没有关键的消融实验被遗漏？例如只去掉模块A但保留了模块A生成的特征？
7. **计算成本是否合理？** — 方法的性能提升是否以极高的计算成本为代价？如果是，这个"性价比"值得吗？
8. **泛化性如何？** — 方法在论文报告之外的数据/场景下还能工作吗？
9. **与同期工作的关系？** — 有没有同时期的类似工作？本文的差异化优势是否真的独特？
10. **可复现性？** — 论文提供了足够的信息让其他人复现吗？没提供代码/数据的有多大影响？
11. **长远价值？** — 5年后回头看，这篇论文更可能是一个milestone还是一个被遗忘的trick？

**写作要求**：
- 不要求11个角度全部覆盖（有些角度可能不适用于某类论文），但至少覆盖其中5-7个
- 每个批判要有具体的理由和证据，不要空泛地说"方法不够好"
- 批判不是否定，建设性地指出改进可能
- 如果你是审稿人，你的"Overall Rating"会是什么？（Strong Accept / Accept / Borderline / Reject）

**HTML呈现方式**：每个角度一小段 + alert样式块呈现总体评价

---

### 维度10：可视化总结

**目标**：一张图胜过千言万语——用HTML/CSS构造论文的核心视觉概念。

**必须包含至少一项（两项更佳）：**

**(A) 方法流程图**
- 使用CSS flexbox/grid + 箭头字符构造方法的端到端流程
- 每个节点代表一个关键组件，节点内标注组件名称
- 箭头使用 `→` `↓` `⟹` 等Unicode字符
- 移动端自动切换为纵向排列

**(B) 核心对比表**
- 本文方法 vs 最相关的现有方法的特性对比
- 行=不同方法，列=不同维度/特性（有/无这种二值标注 + 关键差异简述）
- 本文方法列高亮

**(C) 实验数据图（可选但加分）**
- 使用CSS柱状图/条形图呈现关键实验数据
- 或使用CSS绘制的雷达图呈现多维度评估
- 实验数据的"大数字"呈现（指标名称+数值+单位，大字号突出）

**(D) 概念示意图（如有明确概念）**
- 例如对抗样本、注意力机制、图结构等可以用CSS布局抽象表示

**HTML呈现方式**：纯CSS图表（.flow / .bar-chart / .contrib-grid 类），不允许使用外部图片

---

### 维度11：相关资源

**目标**：让读者一站式获取所有相关材料。

**必须列出**：
- 论文中引用的最重要的参考文献（至少5篇），附带一句话说明每篇与本文的关系
- 官方代码仓库URL（如有）
- 数据集下载链接（如有）
- 项目主页（如有）
- 作者后续相关工作的论文标题（如有）
- 相关教程/博客/解读（如果可搜索到）

**HTML呈现方式**：编号列表 + 可点击的链接 + tag标签标注资源类型

---

### 维度12：可复现性提示

**目标**：如果读者想复现这篇论文，需要注意什么？

**必须覆盖**：
- 关键超参数清单（表格形式）
- 训练/推理的硬件和软件环境要求
- 作者是否开源了代码？如果没有，哪些细节不足以复现？
- 已知的复现陷阱（例如：某个trick对结果影响极大但论文没强调）
- 训练时间和计算资源估算
- 如果论文提供了预训练权重，给出获取方式

**HTML呈现方式**：超参数表格 + 环境信息代码块 + warning框提示注意点

## 第三阶段：生成 HTML 报告

### 3.1 读取模板并借鉴外部资源

1. **先 Read `assets/template.html`**，作为HTML骨架起点
2. **应用第一阶段收集到的Skill资源**：
   - 如果有其他skill提供了更优的图表样式、布局模式、配色方案 → 融合到输出HTML中
   - 如果有代码高亮或公式渲染的最佳实践 → 使用它

## 常见错误与纠正

| 常见错误 | 纠正方法 |
|---------|---------|
| 方法论只写了架构名称，没解释内部机制 | 逐一拆解每个模块的输入输出和设计动机 |
| 批判性思考变成了"我觉得不好" | 每个批判必须附上具体的理由和证据 |
| 实验结果只贴了原始数字，没有分析 | 必须解释数字的含义，为什么好/不好 |
| 完全省略了维度11（资源）和维度12（可复现性） | 即使论文没提供，也要写明"论文未提供代码/数据" |
| 流程图为空或只是一个箭头 | 至少要有3个以上的节点，涵盖输入到输出的全流程 |
| HTML中引用了外部CDN的CSS/字体 | 全部去掉，改为系统字体和内联样式 |
| 公式用纯文本写成一坨，不可读 | 放入 `<pre><code>` 块，用空格和换行格式化 |
| 模板的 `__PLACEHOLDER__` 没有全部替换 | 生成HTML后检查是否还有残留的 `__` 双下划线占位符 |
