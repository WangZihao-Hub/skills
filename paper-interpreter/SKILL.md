# Paper Interpreter — 学术论文深度解读 + 发布

## 概述

本技能提供三种工作模式，**默认为模式 A**：

- **模式 A（解读 + 发布，默认）**：获取论文内容 → 12维度深度解读 → 生成 HTML → 推送到 GitHub Pages → 更新 README
- **模式 B（仅解读）**：获取论文内容 → 12维度深度解读 → 生成 HTML 输出（用户明确说"不用发布"/"只解读"时使用）
- **模式 C（通用发布）**：将已有 HTML 文件发布为 GitHub Pages 网页

## 模式决策

| 用户意图 | 选用模式 |
|----------|----------|
| 解读/分析/总结论文、文章（无论是否提及"发布"） | **模式 A**（默认） |
| 用户明确说"不用发布"、"不需要推送"、"只解读不发布" | 模式 B |
| 用户要求把已有 HTML 文件发布上网 | 模式 C（见末尾） |

# 模式 A & B 共用：12维度深度解读

## 第一阶段：准备工作

### 1.1 确认论文内容来源

- **用户直接粘贴全文**（最常见）→ 直接进入解读流程
- **用户提供PDF文件** → 使用 `pdf` skill 提取文本
- **用户提供URL/DOI/链接** → 使用 WebFetch 获取内容，配合 WebSearch 补充元数据
- **用户只给了标题** → 搜索公开摘要；**必须告知用户**"基于公开摘要解读，深度有限，建议提供全文"
- **用户给了不完整片段** → 明确告知缺失了哪些关键部分，同时基于现有内容给出初步分析

### 1.2 收集环境资源

在开始解读前，扫描当前可用的所有 Skill，判断哪些可以增强输出：
- PDF skill → 提取PDF文本
- guizang-ppt-skill → 借鉴其大字报、网格布局设计
- 其他可视化相关 skill → 借鉴图表样式

### 1.3 判断论文类型

| 论文类型 | 解读侧重 |
|---------|---------|
| **方法型**（ML/CV/NLP） | 方法论和实验是重点，批判性关注泛化性 |
| **系统型**（Systems/DB/OS） | 架构设计是重点，关注工程trade-off |
| **理论型**（ML Theory/Math） | 公式推导逻辑是重点，关注假设合理性 |
| **经验/调查型**（Survey/Benchmark） | 实验设计公平性和覆盖面 |
| **应用型**（Bio/Med/Geo+AI） | 领域问题要讲清楚，关注跨学科意义 |

### 1.4 设定解读深度

| 用户意图 | 深度 |
|----------|------|
| "大概看看"/"扫一眼" | 快速概览：每个维度2-4句话 |
| "解读"/"分析"（默认） | 标准解读：每个维度2-4段 |
| "精读"/"组会汇报" | 深度精读：每个维度详细展开 |

## 第二阶段：12维度深度解读

### 维度1：元信息
- 完整标题、所有作者及机构
- 发表年份、会议/期刊全称、CCF级别/SCI分区
- DOI链接、arXiv ID、是否获奖、引用量
- **HTML**：hero区域集中展示

### 维度2：研究背景与动机
- 所属具体子领域（精确到细粒度）
- 领域演进脉络（2-4个里程碑工作）
- 现存2-3个核心痛点
- 本文针对的是哪个痛点
- **写作**：总→分→总结构，避免泛泛而谈
- **HTML**：段落 + 时间线或痛点列表

### 维度3：核心问题/目标
- 一句话精确陈述核心研究问题
- 子问题逐一列出、研究假设、预期目标
- 区分"研究问题" ≠ "方法"
- **HTML**：醒目的问题陈述块 + 编号列表

### 维度4：方法论/技术路线 ⭐（最详细维度）

**(A) 总体架构概述** — 整体框架 + 数据流 + 训练vs推理区别
**(B) 核心组件逐一拆解** — 每个模块的作用、输入输出、设计动机
**(C) 关键公式** — 核心公式（≤5个），每个解释含义和符号
**(D) 训练策略** — 优化器、学习率、batch size、数据增强、预训练来源
**(E) 创新点对比** — 表格列出本文 vs 2-3个现有方法的差异
**(F) 算法伪代码** — `<pre><code>` 块，中文注释 + 关键步骤
- **HTML**：CSS架构流程图 + 组件分块 + 公式代码块 + 对比表格

### 维度5：实验设计与结果

**(A) 实验设置** — 数据集、评估指标、基线模型、硬件、框架
**(B) 主要结果** — 对比表格（标注最优/次优），分析提升是否显著
**(C) 消融实验** — 每个消融的设置/目的/结论，哪个组件最关键
**(D) 参数敏感性** — 关键超参数的影响
**(E) 可视化分析** — 定性分析的发现
- **HTML**：对比表格 + CSS柱状图 + 关键数字大字报

### 维度6：主要贡献
- 3-5条要点，区分核心/次要、理论/实践贡献
- **HTML**：卡片网格（contrib-grid）

### 维度7：讨论与局限性
- 作者承认的所有局限性、严重程度评估
- 方法适用范围边界
- **HTML**：warning样式的alert框

### 维度8：未来工作建议
- 作者提出的后续方向
- 基于局限性的合理延伸
- 读者最值得做的1-2个后续工作
- **HTML**：编号列表 + 每条理由

### 维度9：批判性思考 ⭐（区分"会读"和"只会翻译"）

逐一检查11个角度（至少覆盖5-7个）：
1. 动机是否成立？2. 假设是否合理？3. 是否过拟合特定设置？
4. 对比是否公平？5. 提升是否显著？6. 消融是否充分？
7. 计算成本是否合理？8. 泛化性如何？9. 与同期工作的关系？
10. 可复现性？11. 长远价值？

- 每个批判要有具体理由和证据
- 给出审稿人 Overall Rating（Strong Accept / Accept / Borderline / Reject）
- **HTML**：每角度一段 + alert块呈现总体评价

### 维度10：可视化总结 ⭐

至少包含一项（两项更佳）：
- **(A) 方法流程图** — CSS flexbox/grid + Unicode箭头，移动端纵向
- **(B) 核心对比表** — 本文 vs 现有方法特性对比
- **(C) 实验数据图** — CSS柱状图/条形图
- **(D) 概念示意图** — 用CSS布局抽象表示核心概念
- **纯CSS图表，不允许外部图片**

### 维度11：相关资源
- 最重要的参考文献（≥5篇）+ 与本文的关系
- 官方代码仓库、数据集链接、项目主页
- 相关教程/博客
- **HTML**：编号列表 + 可点击链接 + tag标签

### 维度12：可复现性提示
- 关键超参数清单（表格）
- 硬件和软件环境要求
- 是否开源代码？哪些细节不足？
- 已知的复现陷阱、训练时间估算
- **HTML**：超参数表格 + 环境信息代码块 + warning框

## 第三阶段：生成 HTML 报告

### 3.1 样式系统

| CSS类 | 用途 | 使用场景 |
|-------|------|---------|
| `.hero` | 顶部标题区 | 维度1 |
| `.toc` | 粘性导航 | 12维度快速跳转 |
| `section` | 白色卡片 | 每个维度容器 |
| `.contrib-grid` / `.contrib-card` | 贡献卡片 | 维度6 |
| `.flow` / `.node` / `.arrow` | CSS流程图 | 维度4、10 |
| `.bar-chart` / `.bar-col` / `.bar` | CSS柱状图 | 维度5 |
| `table` / `.table-wrap` | 数据表格 | 维度4、5、12 |
| `blockquote` | 引用块 | 原文摘录 |
| `pre > code` | 代码/公式块 | 维度4、12 |
| `.alert.info` | 蓝色提示 | 维度3 |
| `.alert.warning` | 黄色警告 | 维度7、12 |
| `.alert.success` | 绿色正面 | 维度6 |
| `.tag` | 标签徽章 | 维度2、11 |

### 3.2 内容填充规则

- `<title>` = `[论文解读] {论文标题}`
- 公式用 `<pre><code>` + `<i>`/`<sup>`/`<sub>` + Unicode数学符号（不用MathJax）
- 表格用 `<div class="table-wrap">` 包裹，斑马纹
- 论文原文关键句用 `<blockquote>`
- 所有CSS/JS内联，**零外部依赖**

### 3.3 响应式设计

- 桌面端（≥768px）：多栏布局，图表横向
- 移动端（<768px）：单栏堆叠，流程图纵向，表格横向滚动
- 打印：隐藏导航，去阴影，黑白可读

## 模式 A 专属：发布到 GitHub Pages

### 固定路径

| 项目 | 值 |
|------|-----|
| 仓库根目录 | `C:\Users\WZH\Desktop\documents` |
| 论文目录 | `paper/<slug>/` |
| HTML 文件名 | `index.html`（固定） |
| README | `README.md` |
| GitHub 远程 | `git@github.com:WangZihao-Hub/Document.git` |
| Pages 根 URL | `https://wangzihao-hub.github.io/Document/` |

### A-1：确定 slug

根据论文标题生成 kebab-case 短名：`<核心关键词>-<第一作者姓氏><年份>`

示例：
- "Attack vulnerability of scale-free networks due to cascading failures" → `cascading-failure-wang2008`
- "Deep Residual Learning for Image Recognition" → `resnet-he2016`

原则：简短、可读、全小写英文和数字、连字符分隔。**确定后展示给用户确认再创建目录。**

### A-2：保存 HTML 并推送

```bash
cd "C:\Users\WZH\Desktop\documents"
git add "paper/<slug>/index.html"
git commit -m "Publish paper/<slug>/index.html - <论文简称>"
git push
```

### A-3：更新 README.md

1. Read `README.md` 查看当前结构
2. 在 `### 📝 Paper（论文）` 小节末尾追加：

```markdown
- **[<论文中文简称>](./paper/<slug>/)** — [网页版](https://wangzihao-hub.github.io/Document/paper/<slug>/) · <一句话描述 | 第一作者 et al. (年份)>
```

3. 提交推送 README 更新

### A-4：输出总结

- 📄 论文标题及作者
- 📁 本地文件路径
- 🔗 在线访问链接：`https://wangzihao-hub.github.io/Document/paper/<slug>/`
- 📊 报告亮点

### A-注意事项

- 如果论文仅基于摘要解读，报告中标注"基于公开摘要解读，深度有限"
- slug 确定后展示给用户确认
- 推送前检查未提交变更
- README 追加条目格式与已有条目一致
- 目标文件夹已存在 `index.html` 时，询问用户是否覆盖

# 模式 C：通用 HTML 发布

将任意本地 HTML 文件发布为 GitHub Pages。

### C-1：确认文件 → C-2：重命名为 index.html → C-3：提交推送 → C-4：输出链接

```bash
mv "原始文件名.html" "所在目录/index.html"
cd "<repo-root>" && git add "<path-to-index.html>" && git commit -m "Publish <folder>/index.html" && git push
```

目标 URL = `https://<user>.github.io/<repo>/<子目录>/`

前提：仓库已开启 GitHub Pages（Settings → Pages → 选择分支）。

## 质量自检清单

在输出HTML之前，逐项确认：

- [ ] 12个维度全部覆盖
- [ ] 维度4（方法论）详细程度明显高于其他维度
- [ ] 维度5（实验）至少包含一个对比表格
- [ ] 维度6（贡献）使用卡片网格布局
- [ ] 维度7（局限性）列出作者承认的每条局限
- [ ] 维度9（批判性）覆盖至少5个批判角度，有具体论据
- [ ] 维度10（可视化）至少包含一个CSS流程图或对比表格
- [ ] 维度11（资源）列出至少3条参考文献
- [ ] 维度12（可复现性）列出关键超参数
- [ ] HTML中 `<title>` 正确设置
- [ ] 零外部URL依赖（无CDN CSS/JS/字体）
- [ ] 所有CSS/JS内联
- [ ] 包含 `<meta viewport>` 标签
- [ ] 包含 `@media (max-width: 768px)` 响应式
- [ ] 包含 `@media print` 打印样式
- [ ] 所有 `id` 锚点与目录 `href` 匹配
- [ ] 无残留 `__PLACEHOLDER__` 占位符

## 论文信息不完整时的降级策略

| 情况 | 策略 |
|------|------|
| 只有标题和摘要 | 生成简化版，每个维度标记"基于摘要推断" |
| 缺少实验细节 | 写明"论文原文中实验细节未充分描述" |
| 非英文论文 | 保留原文关键术语，解释用中文 |
| 论文内容有矛盾 | 指出矛盾（在维度9中），给出合理推断 |
